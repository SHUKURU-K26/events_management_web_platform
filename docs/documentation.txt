## 📋 Project Overview

SERM is a modern web-based platform designed to replace the inefficient use of spreadsheets and WhatsApp communication in an events and production company. It provides a unified system where:

- **Clients** can have their events booked and tracked by managers
- **Managers** can oversee all operations from a single dashboard
- **Staff** can track their assigned tasks and events in real time
- **Resources** are managed with full availability tracking and double-booking prevention
- **Finances** are tracked with monthly reports exportable as PDF

---

## 🚨 Problem Statement

Before SERM, the company faced the following operational challenges:

- **No centralized event management** — events were tracked manually on spreadsheets
- **Staff coordination via WhatsApp** — assignments were communicated informally with no tracking
- **Resource double-booking** — equipment was sometimes assigned to multiple events simultaneously
- **No financial visibility** — expenses and revenue were recorded inconsistently across different files
- **No client booking system** — client requests were handled verbally with no formal tracking
- **No reporting** — generating monthly or periodic reports required manual compilation of data
- **No role-based access** — all staff had access to the same information regardless of their role

---

## ✅ Solution

SERM addresses all the above challenges by providing:

- A **centralized dashboard** with real-time stats for managers and staff
- **Role-based access control** — superadmin, manager, and staff roles with different permissions
- **Event management** with staff assignment and status tracking
- **Resource management** with availability tracking and automatic double-booking prevention
- **Client booking system** managed entirely by managers
- **Financial tracking** with monthly PDF report export
- **In-app notifications** for task and event assignments
- **Task management** with progress tracking for staff

## 🚀 Setup Instructions

### Prerequisites
- Node.js 18+
- MySQL / MariaDB
- pnpm (recommended) or npm

### 1. Clone the Repository
```bash
git clone https://github.com/SHUKURU-K26/events_management_web_platform.git
cd events_management_web_platform
```

### 2. Setup Database
```bash
# Login to MySQL
mysql -u root -p

# Create database
CREATE DATABASE serm;
USE serm;

# Run the schema (create all tables as documented above)
```

### 3. Setup Backend
```bash
cd server
pnpm install

# Create .env file
cp .env.example .env

# Fill in your .env values:
# DB_HOST=127.0.0.1
# DB_USER=root
# DB_PASSWORD=yourpassword
# DB_NAME=serm
# JWT_SECRET=your_secret_key_here
# PORT=3000

# Start development server
pnpm dev
```

### 4. Setup Frontend
```bash
cd client
pnpm install

# Start development server
pnpm dev
```

### 5. Access the Application
```
Frontend: http://localhost:5173
Backend:  http://localhost:3000
```

## 🌍 Deployment Guide

### Backend — Render

1. Push code to GitHub
2. Go to [render.com](https://render.com) → New Web Service
3. Connect your GitHub repository
4. Set:
   - **Root Directory:** `server`
   - **Build Command:** `pnpm install`
   - **Start Command:** `node server.js`
5. Add environment variables from your `.env`
6. Deploy!

### Frontend — Vercel

1. Go to [vercel.com](https://vercel.com) → New Project
2. Import your GitHub repository
3. Set:
   - **Root Directory:** `client`
   - **Framework:** Vite
4. Add environment variable:
   - `VITE_API_URL` = your Render backend URL
5. Deploy!

### Database — Railway

1. Go to [railway.app](https://railway.app) → New Project
2. Add MySQL service
3. Copy the connection details to your Render environment variables

## 🏗️ Architecture

```
┌─────────────────┐         ┌─────────────────┐         ┌─────────────────┐
│                 │  HTTP   │                 │  SQL    │                 │
│   React + Vite  │ ──────► │  Express + Node │ ──────► │  MySQL Database │
│   (Frontend)    │ ◄────── │  (REST API)     │ ◄────── │                 │
│                 │  JSON   │                 │  Data   │                 │
└─────────────────┘         └─────────────────┘         └─────────────────┘
     Vercel                      Render                     Railway
```

### Key Architecture Decisions

- **JWT Authentication** — stateless token-based auth for scalability
- **Role-based Access Control** — middleware-level permission enforcement
- **Connection Pooling** — MySQL connection pool for performance
- **Junction Tables** — many-to-many relationships for staff/event/task assignments
- **Nullable Foreign Keys** — tasks can exist independently of events
- **Auto Status Updates** — resources auto-return when events complete or bookings cancel

---

## 💡 Assumptions Made

1. The company operates in Rwanda — currency is RWF (Rwandan Franc)
2. Only managers and superadmins can create/manage events and bookings — clients do not access the system directly
3. A resource can only be assigned to one event at a time — single assignment model per serial number
4. Staff members can update their own task progress but cannot create or delete tasks
5. Notifications are in-app only — no email notifications implemented in this version
6. The system assumes a single timezone — UTC is used throughout

---

## 🧗 Challenges Faced

1. **Timezone date shifting** — MySQL dates were being shifted by timezone offset when returned to the frontend. Fixed by adding `timezone: '+00:00'` to the MySQL connection pool config and using `.slice(0, 10)` for date display.

2. **React input focus loss** — Defining form components inside the parent component caused React to remount them on every keystroke, losing input focus. Fixed by moving all reusable components (Modal, StatusBadge, CategoryBadge) outside the main component.

3. **JWT token payload inconsistency** — The login route was storing the user ID as `id` while routes were reading `user_id`, causing staff-specific queries to fail silently. Fixed by standardizing the JWT payload to always use `user_id`.

4. **Many-to-many relationship design** — Determining the correct junction table structure for staff-to-event and staff-to-task assignments required careful planning to avoid data redundancy while supporting multiple assignments.

5. **Resource double-booking prevention** — Implementing a status-based availability system that automatically updates resource status on assignment and returns resources when events complete or bookings are cancelled.

6. **Role-based dashboard** — Staff and managers needed different dashboard metrics. Solved by adding role detection in both the backend dashboard route and frontend rendering logic.
